# 修复 CLI 缺失数据版本检查

## 诊断回顾

`check_data_version` 函数（`commands.rs:102-129`）存在且有完整测试，但生产代码里**只有 GUI 的 `init` 命令调用它**（`commands.rs:411`）。CLI 的 `cli::run()`（`cli/mod.rs:145-240`）及所有子命令（entries/commitments/dimensions）完全不读 `version.txt`。后果：version.txt 被提升后，老版本 CLI 会用老格式逻辑静默读写，造成数据污染且无任何报错。设计文档 `2026-07-04-data-version-design.md` §3.1 把 version.txt 的读取者仅定义为 `commands::init`，CLI 读写命令是设计盲区。

## 实施方案

### 1. 新增辅助函数 `ensure_compatible_version`

**文件**：`src-tauri/src/cli/mod.rs`

新增函数（放在 `run()` 之前），职责：判断当前命令是否需要版本检查（`migrate` 豁免），需要则调用 `check_data_version`，把 `InitResult` 错误变体翻译成用户可读的错误字符串。

```rust
/// 检查数据目录版本是否兼容当前代码。
/// `migrate` 子命令豁免——它本身可能就是来升版本的。
/// 返回 Err(msg) 时，调用方应打印 msg 并 exit(1)。
fn ensure_compatible_version(
    root: &std::path::Path,
    command: &Commands,
) -> Result<(), String> {
    use crate::commands::check_data_version;
    use crate::models::{CURRENT_DATA_VERSION, InitResult};

    // migrate 豁免：它是版本升级工具，读旧版本、写新版本是正常行为
    if matches!(command, Commands::Migrate) {
        return Ok(());
    }

    match check_data_version(root, CURRENT_DATA_VERSION) {
        Ok(()) => Ok(()),
        Err(InitResult::DataVersionNotFound { root_path }) => Err(format!(
            "Data version file not found in {}.\n\
             Run `logbook-cli migrate` to initialize the data version, \
             or start the Logbook GUI once to set up.",
            root_path
        )),
        Err(InitResult::DataVersionMismatch { root_path, expected, found }) => Err(format!(
            "Data format version mismatch in {}.\n\
             Expected version {}, found version {}.\n\
             This CLI (version {}) expects data version {}. \
             Run `logbook-cli migrate` with a newer build, or update Logbook.",
            root_path, expected, found,
            env!("CARGO_PKG_VERSION"), expected
        )),
        // check_data_version 只返回上述两种 Err 变体
        Err(other) => unreachable!("check_data_version returned unexpected variant: {:?}", other),
    }
}
```

**为何抽成函数**：`run()` 里用 `std::process::exit`，无法单测。抽成返回 `Result` 的纯函数后，migrate 豁免逻辑和错误信息格式化都可单测。

### 2. 在 `run()` 中调用

**文件**：`src-tauri/src/cli/mod.rs`，`run()` 第 199 行（`eprintln!("Using data root...")`）之后、第 201 行（`match cli.command`）之前插入：

```rust
eprintln!("Using data root: {}", root.display());

// 版本检查：防止老 CLI 写坏新格式数据，或读到残缺结果。
// migrate 豁免（见 ensure_compatible_version）。
if let Err(msg) = ensure_compatible_version(&root, &cli.command) {
    crate::error_log::log_error("cli", &msg);
    eprintln!("Error: {}", msg);
    std::process::exit(1);
}

match cli.command {
    // ... 原有分发逻辑不变
```

**插入位置理由**：此时 root 已解析（第 190 行）、锁已处理（第 162-188 行），命令即将分发。版本检查是一道独立的"准入门"，放在 root 解析之后、分发之前最清晰。实例锁顺序无所谓——写命令已持锁，exit 时自动释放；只读命令本就不持锁。

### 3. 错误信息走 stderr，不特殊处理 `--json`

与现有实例锁错误（`cli/mod.rs:170-182`）和 `print_error`（`cli/output.rs:17`）保持一致：致命错误走 `eprintln!`（stderr），正常输出走 stdout。`--json` 模式下版本不匹配仍是致命错误，走 stderr 文本。这样 stdout 永远只有成功结果，便于管道脚本判断。

### 4. 单元测试

**文件**：`src-tauri/src/cli/mod.rs` 的 `#[cfg(test)] mod tests`（已存在，第 242 行起）。

新增测试覆盖 `ensure_compatible_version`：
- `migrate 命令始终豁免`（无论 version.txt 是什么状态都返回 Ok）——用任意 version.txt 状态 + `Commands::Migrate` 验证
- `version 匹配 → Ok`
- `version 不匹配 → Err，信息含 expected/found`
- `无 version.txt → Err，信息含 not found`

测试用临时目录 + `files::write_version_file`，模式参照现有 `tests/data_version_integration.rs`。`Commands` 构造方式参照 `cli/mod.rs` 已有测试（第 249-310 行）。

### 5. 不做的事

- **不改 `migrate.rs:122` 的硬编码 `2`**。理由：migrate 写的是"迁移目标版本"（它只实现了 v1→v2），语义上 ≠ `CURRENT_DATA_VERSION`。若未来代码 bump 到 v3 但 migrate 未扩展，migrate 仍应写 2（它只保证迁到 v2）。直接换成常量反而会写出错误版本号。这个点作为独立风险记录，不在本次修复范围。（用户第一轮选择了"规划 CLI 版本检查修复"，未选含 migrate 修复的选项。）
- **不加 CLI 版本检查的集成测试（跑 `run()` 全流程）**。`run()` 含 `process::exit`，集成测试无法捕获。抽函数 + 单测已覆盖核心逻辑，足够。
- **不改 `check_data_version` 本身**。它已经是纯函数且测试充分，CLI 只是新增一个调用点。

## 影响面

- **行为变化**：任何 CLI 命令（除 migrate）在 version.txt 缺失或不匹配时，从"静默执行"变为"报错 exit(1)"。当前 `CURRENT_DATA_VERSION = 2`，已 setup 或已 migrate 的用户 version.txt 都是 2，不受影响。受影响的是：用老 CLI（编译时 version≠2 的历史二进制）操作 v2 数据，或未来版本提升后老 CLI 操作新数据——正是要修的场景。
- **文件改动**：仅 `src-tauri/src/cli/mod.rs`（加一个函数 + run 里加几行 + 测试）。
- **现有测试**：不受影响，全部继续通过。`tests/data_version_integration.rs` 测的是 `check_data_version` 函数本身，本次只新增调用点。

## 遗留风险（记录，本次不处理）

`migrate.rs:122` 硬编码 `2`。语义正确（迁移目标版本），但缺乏注释说明"为何是字面量而非常量"。建议后续加一行注释，或引入 `MIGRATION_TARGET_VERSION` 命名常量提升可读性。不影响本次修复正确性。
